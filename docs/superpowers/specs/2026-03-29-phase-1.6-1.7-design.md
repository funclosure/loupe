# Phase 1.6 + 1.7 Design — Server-side File I/O & Lens Brevity

## Overview

Two focused changes: (1) move all file I/O through the Bun server, eliminating the browser File System Access API, and (2) add a shared brevity instruction to lens sessions so responses default to concise.

---

## Part 1 — Server-side File I/O

### Problem

When `loupe sample.md` opens a file, the server reads it but the client has no way to save back — the File System Access API operates in a separate browser sandbox. The `?file=path` query param is stripped on load, so refresh disconnects from the file entirely. Opening via browser picker gives a handle, but it's lost on refresh.

### Design

**Server tracks the active file path.** The server resolves and stores the path when a file is opened. All reads and writes go through server endpoints.

#### New/Modified Server Endpoints (`routes.ts`)

| Endpoint | Method | Behavior |
|---|---|---|
| `GET /api/file?path=...` | GET | Read file from disk. Returns `{ content, filename, path }`. Returns 404 if file doesn't exist. Stores resolved path as active file on server. |
| `POST /api/file` | POST | Write `{ content, path? }` to disk. If `path` is provided, resolve it relative to CWD, create the file if needed, set it as active, and write. If `path` is omitted, write to the current active path. Returns `{ ok: true, path, filename }`. Returns 400 if no active file and no path given. |
| `GET /api/files` | GET | List `*.md` files in CWD (flat, non-recursive, alphabetically sorted). Returns `{ files: [{ name, path }] }`. |

The server holds the active file path in memory (on `RouteHandler` or a small `FileStore` class). Only one file is active at a time — opening a new file replaces the previous path. This assumes single-tab usage; multiple tabs would share/overwrite the active path.

#### Server Index Changes (`index.ts`)

- Pass CWD to `RouteHandler` (or derive from `process.cwd()`) for file listing.

#### CLI Changes (`cli.ts`)

- No changes to `cli.ts`. `?file=path` still passed as query param. Note: since the URL is no longer stripped, `?file=sample.md` remains visible in the browser address bar — this is intentional.

#### Client `use-file.ts` — Rewrite

Remove all File System Access API code (`showOpenFilePicker`, `showSaveFilePicker`, `FileSystemFileHandle`, `createWritable`). Replace with server-backed operations:

```typescript
interface UseFileReturn {
  filename: string;
  initialContent: string;
  saveState: "saved" | "saving" | "unsaved";
  openFile: () => void;           // Opens file picker modal
  save: () => Promise<void>;      // POST /api/file
  saveFileAs: () => void;         // Opens save-as prompt
  updateContent: (content: string) => void;
  filePath: string | null;        // Server-side path (null = untitled)
  isFilePickerOpen: boolean;
  setFilePickerOpen: (open: boolean) => void;
}
```

**Save flow:**
- `updateContent()` still debounces to localStorage (500ms) as crash safety
- `updateContent()` also debounces `POST /api/file` (1000ms) when `filePath` is set — auto-save to disk
- `save()` cancels any pending debounce and forces an immediate `POST /api/file`
- If no `filePath` (untitled), `Cmd+S` opens a save-as prompt (text input for filename, saves relative to CWD via `POST /api/file { content, path }`)
- On server write failure (permissions, disk full), set `saveState` to `"unsaved"` — no toast, just the indicator

**Open flow:**
- `Cmd+O` opens a file picker modal showing `*.md` files from `GET /api/files`
- Selecting a file calls `GET /api/file?path=<selected>` and loads content into editor
- This also sets `filePath` on both client and server

**CLI open flow:**
- `?file=path` in URL → client calls `GET /api/file?path=...` on mount (same as today)
- **Do NOT strip `?file=` from URL** — keep it so refresh reconnects to the same file
- Server stores path, client stores filename and filePath

#### Client `App.tsx` Changes

- Remove `window.history.replaceState({}, "", "/")` — preserve `?file=` across refresh
- Wire `Cmd+O` to `setFilePickerOpen(true)` instead of browser picker
- Wire `Cmd+S` to `save()` (immediate server write) or save-as prompt if untitled

#### Client `TopBar.tsx` Changes

- `saveState` no longer needs `"unavailable"` — server save is always available when a file is open
- Remove "download to save" message
- Show "saved" briefly after server confirms write

#### File Picker Modal (`src/client/chrome/FilePicker.tsx`)

Reuse the `LensPicker` modal pattern — a simple overlay listing `.md` files from CWD:

- `GET /api/files` on open
- Each item shows filename, click to open
- Escape or click-outside to dismiss
- If no `.md` files found, show empty state

#### localStorage Role

Unchanged — always backs up content with 500ms debounce. On mount:
1. Check `?file=path` → load from server if present
2. Otherwise, restore from localStorage (crash recovery / scratch pad)

## Part 2 — Lens Brevity Instruction

### Problem

Lens responses are too verbose. The current closing instruction ("Keep responses concise and conversational. You are a companion, not a lecturer.") doesn't constrain the model enough.

### Design

Replace the closing instruction in `LensSession.buildSystemPrompt()`:

**Before:**
```
Keep responses concise and conversational. You are a companion, not a lecturer.
```

**After:**
```
Respond in 1–3 sentences. Be direct — no preamble, no bullet lists, no summaries.
If the writer asks you to elaborate or go deeper, then expand fully.
You are a writing companion, not an essay generator.
```

This is the only change — appended to every lens system prompt in `lens-session.ts`. Individual lens preset prompts in `lens-presets.ts` stay unchanged.
